# Architecture: High-Speed ADC Signal Acquisition Pipeline

> **Target hardware:** Digilent USB104 A7 Rev. B.2 (XC7A100T-1CSG324I) + Zmod ADC 1410-105 (AD9648)
> **Host link:** On-board FT232H DPTI (synchronous FIFO, ~7 MB/s sustained empirical, ~30 MB/s theoretical)
> **Last updated:** 2026-06-30

> **Design principle:** Debug observability is a first-class architectural property, not an afterthought.
> Every pipeline block exposes a standardised debug interface. The audit aggregator
> collects all debug signals onto a DPTI-accessible register map. Built-in self-test
> (BIST) validates every block from host without external equipment. See §12.

## Purpose

This system captures high-speed analog signals from High‑Purity Germanium (HPGe) detectors via the **AD9648** dual 14-bit ADC (Zmod ADC 1410-105) on a **Digilent USB104 A7** carrier, buffers raw waveform data in the **XC7A100T** FPGA block RAM, and streams selected events to a host PC over the on-board **FT232H DPTI** synchronous FIFO interface at up to ~30 MB/s.

It is designed specifically for nuclear/particle physics detector readout—targeting the characteristic signal shapes, time constants, and energy resolution requirements of HPGe detectors—while remaining adaptable to other high-rate pulse acquisition applications.

## Clock Domains

| Domain      | Source              | Frequency  | Used by                                      |
|-------------|---------------------|:----------:|----------------------------------------------|
| `adc_dco`   | AD9648 DCO output   | ~105 MHz   | ADC interface, deinterleaver, upstream of CDC |
| `sys_clk`   | Onboard 100 MHz oscillator (BUFG-buffered, no PLL) | 100 MHz | Everything downstream of CDC FIFO       |
| `ft_clk`    | FT232H `prog_clko`  | 60 MHz     | FT232H DPTI bridge (TX FIFO read side)       |

**Note:** `adc_dco` is generated by the AD9648 as a clock output synchronous to its data. It must be connected to a global clock buffer (IBUFG/BUFG). **`sys_clk` uses the on-board 100 MHz oscillator directly (BUFG-buffered, no PLL/MMCM).** This matches the proven sig_recorder implementation (`top.v` uses `wire clk_100 = clk` with the onboard osc). The 100 MHz osc and 105 MHz DCO cross via the CDC FIFO, which is the designed crossing point. A PLL is optional (for synchronous sys_clk from adc_dco) but not required and adds an unnecessary failure mode (MMCM lock).

**Rule:** No signal may cross a clock domain boundary without passing through the CDC FIFO or an explicit synchroniser. The CDC FIFO is the single authorised crossing point between `adc_dco` and `sys_clk`.

## Module Descriptions

### 1. ADC Interface

- **Receives:** 14-bit parallel CMOS bus from AD9648 (`adc_data[13:0]`), single-ended LVCMOS18, plus `adc_dco` (data clock output from ADC)
- **Produces:** Deinterleaved channel pair: `ch_a_data[13:0]` (rising edge), `ch_b_data[13:0]` (falling edge), plus `data_valid` strobe
- **Clock domain:** `adc_dco`
- **Debug interface (§12.1):** DCO lock status, bit-error count, channel A/B sample continuity count, IDDR calibration state

**The AD9648 in interleaved CMOS mode** — Unlike a traditional LVDS SERDES interface, the AD9648 in CMOS mode presents both channels on a single 14‑bit data bus. Channel A is sampled on the **rising edge** of DCO, Channel B on the **falling edge**. This is standard DDR-style data output.

```
adc_dco   ─┬──┬──┬──┬──┬──┬──┬──┬──
           │  │  │  │  │  │  │  │  │
adc_data  X A0 X B0 X A1 X B1 X A2 X
           │  │  │  │  │  │  │  │  │
           rising   falling  rising
```

**Implementation:**

```systemverilog
// IDDR primitive captures both edges
IDDR #(.DDR_CLK_EDGE("SAME_EDGE_PIPELINED")) iddr_inst (
    .Q1(ch_a_data),  // rising edge
    .Q2(ch_b_data),  // falling edge
    .C(adc_dco),
    .CE(1'b1),
    .D(adc_data),
    .R(1'b0),
    .S(1'b0)
);
```

**Notes:**
- No SERDES, no IDELAY, no bit-slip — the interface is direct CMOS.
- Input timing constraints are provided in the XDC (see `hardware_spec/USB104_A7_Zmod_ADC1410.xdc`). DCO-to-data skew is 3.24 ns (min) to 5.44 ns (max).
- ADC sample clock (`adc_clk_p/n`) is generated via an ODDR + OBUFDS from a PLL output (differential DIFF_SSTL18_I).
- Configuration of the AD9648 (gain, coupling) is done via the SPI or I²C interface before acquisition starts.

### 2. CDC FIFO

- **Receives:** Deinterleaved sample pair from ADC Interface (`adc_dco` domain)
- **Produces:** Sample pair to Signal Processing (`sys_clk` domain)
- **Responsibilities:** Safe clock domain crossing; absorbs phase/frequency difference between `adc_dco` (which may drift slightly) and the stable `sys_clk` PLL.
- **Implementation:** **One wide async FIFO** using `xpm_fifo_async` (Xilinx Parameterized Macro). Packs both channels into a single 32-bit word (ch_a[15:0] + ch_b[15:0]) per DCO cycle. One FIFO primitive, one CDC domain, simpler debug. `xpm_fifo_async` handles Gray-code pointers and CDC internally — no hand-written pointer logic. Not FIFO Generator IP (requires .xci files) or direct FIFO18E2 (requires hand-written Gray-code pointers).
- **Key parameters:** `FIFO_DEPTH` — power of 2, sized to absorb worst-case phase drift (e.g. 64 entries is sufficient)
- **Timing constraint:** Paths crossing this boundary must be marked false paths or max-delay constrained in XDC.
- **Debug interface (§12.1):** overflow/underflow flags, fill level (instantaneous + watermark), write/read pointer difference, CDC sync stage status

### 3. Signal Processing *(optional)*

- **Receives:** Raw sample stream, dual-channel (`sys_clk`)
- **Produces:** Conditioned sample stream, dual-channel (`sys_clk`)
- **Responsibilities (configurable, may be bypassed, in pipeline order):**
  - **Glitch filter:** 3-stage temporal up-down pattern detector. Removes deterministic AD9648 artifact (`0x0C98` every 256 DCO cycles). Principle: a glitch is UP-then-DOWN (or DOWN-then-UP) in consecutive samples; a real edge moves monotonically. Three separate `always @(posedge clk)` blocks prevent Vivado from inferring SRL32 shift-register optimisation. Threshold configurable (default 500). Verified 100% effective in event mode (0 glitches in 912,600 samples). See *Glitch Artifact Characterization* below.
  - **Debug interface (§12.1):** glitch count (total + since-last-read), max jump detected, bypass mode status, current threshold setting. Each sub-block (glitch filter, baseline estimator, etc.) exposes its own dbg_if when implemented.
  - **Digital filtering:** FIR/IIR to suppress noise (up to ~50 MHz bandwidth at 105 MS/s)
  - **Pole-zero correction:** Compensates for HPGe preamplifier decay tail (typically ~50–200 µs time constant)
  - **Baseline estimation:** First-order IIR (EWMA) running average: `baseline <= baseline + (sample - baseline) >> N`. Resource cost: **~0 DSP slices, <100 LUTs, 1 register per channel**. Alpha shift N=8–12 (time constant ≈ 2.5–40 µs @ 100 MHz — fast enough for thermal drift, slow enough to ignore pulses). Dual-channel feasible with one arithmetic unit + MUX.
  - **Trapezoidal shaping** (optional): Moving-window deconvolution for energy measurement, with adjustable rise time (e.g. 6–12 µs for HPGe)
- **Clock domain:** `sys_clk`
- **Notes:** All stages are pipelined. Latency is fixed and must be accounted for in timestamp generation downstream. For HPGe detectors, the pole-zero time constant should be register‑programmable to match the preamplifier decay.

### 3a. Glitch Artifact Characterization

The AD9648 was found to emit a **deterministic single-sample artifact** with value `0x0C98` (3224 decimal) every **exactly 256 DCO cycles** (2.56 µs). Confirmed by:

- FFT autocorrelation on 1014-event ROOT dataset (period 256.000, autocorr peak at lag 256)
- The artifact first appears at `adc_data_raw` — straight from the AD9648 — before any FPGA logic
- Unaffected by: fan EMI, CDCE6214 config, AD9648 SPI settings, FIFO depth, board power-cycle
- Analysis: the value replaces **any** signal level, not just baseline — confirmed mid-pulse glitches

**Hypothesised root cause:** Internal AD9648 digital pipeline/calibration artifact. Period = 2⁸ (256) suggests an 8-bit internal counter wrapping. The fixed value `0x0C98` is inconsistent with a timing violation (which would produce distributed or metastable values). An alternative theory (sync mismatch between ADC and FPGA data clock) is being investigated — see pending experiments below.

**Current mitigation (HW):** 3-stage up-down pattern detector, threshold = 500 counts (normal slew max = 135 counts/sample, glitch jump = 1376–4968 counts). 100% removal verified in event mode (0/912,600 samples).

**Pending root-cause experiments** (accessible via SPI register writes on existing bitstream):
1. **DCO polarity invert** (reg `0x16[7]` toggle) — if glitch changes channel or frequency → DDR capture issue → sync-mismatch theory supported
2. **Clock divide** (reg `0x0B` set 2/4) — if glitch period scales → internal ADC counter; if stays 256 → fixed pipeline stage
3. **Test patterns** (reg `0x0D` walking 1's) — if glitch superposes → digital backend; if clean → analog/pre-mux path
4. **Output delay sweep** (reg `0x17` 0x00–0xFF) — if glitch disappears → marginal DCO hold time
5. **Non-interleaved CMOS** (reg `0x14[6]=0`) — if glitch disappears → interleave switching artifact

If the three quick experiments (DCO invert, clock divide, test pattern) all point to ADC-internal origin, the sync-mismatch theory is ruled out. A dedicated minimal test project (IDDR → buffer → DPTI, no trigger/buffer complexity) may be warranted for controlled register sweeping.

### 4. Circular Waveform Buffer

- **Receives:** Conditioned sample stream
- **Produces:** Random-read access to any stored sample window (via address + length)
- **Implementation:** Dual-port BRAM. Write pointer increments continuously (wraps). Read port is available to Trigger Logic and Waveform Reader simultaneously.
- **Clock domain:** `sys_clk`
- **Debug interface (§12.1):** live wr_ptr, live burst_rd_ptr, wrap_around_count, max_fill_watermark, collision flag (wr_ptr == rd_addr during active burst)

**XC7A100T BRAM budget:** The FPGA has **4,860 Kb (607 KB) total block RAM**. This is a hard constraint.

| Configuration | Sample Width | Max Depth | Time @ 105 MS/s | Notes |
|---|---|---|---|---|
| Single channel | 16 bits | ~300K samples | ~2.9 ms | All BRAM dedicated to one channel |
| Dual channel, independent | 16 bits each | ~150K samples each | ~1.4 ms | Split evenly |
| Dual channel, packed | 32 bits | ~150K 32-bit words | ~1.4 ms continuous | Both channels in one RAM |
| **Current impl. (Issue 001)** | **16 bits** | **8192 samples** | **78.0 µs** | 3 BRAMs — capture/transport separated |

HPGe pulse widths are typically ~10–100 µs, so even the dual-channel configuration provides >10× the pulse length as pre‑trigger buffer headroom. The current implementation uses an 8192-depth circular buffer (capture memory) plus a separate 4096×16 TX FIFO (transport memory) — the 2-level buffering architecture.

**Event window sizing:** A typical HPGe event window of 1800 samples (600 pre + 1200 post) spans **17.14 µs @ 105 MSPS**. The physics-critical **rising edge (~100–1000 ns = 10–105 samples)** is fully captured. The 600-sample pre-trigger (5.7 µs) provides >5× the longest rise time for baseline characterisation. Both `cfg_pre` and `cfg_post` are parameterizable.

- **Key parameters:** `BUF_DEPTH` (power of 2, up to ~2^18 = 262K for single-channel), `PRETRIGGER_SAMPLES`
- **Pretrigger requirement:** At 105 MS/s, 100 µs of pretrigger = 10,500 samples. The buffer is deep enough for this.
- **Constraint:** Because the buffer is relatively small (~2.9 ms of data), the Waveform Reader must extract events faster than the write pointer wraps. See §9.

### 5. Trigger Logic

- **Receives:** Live sample stream (tapped before or after buffer) + read port of waveform buffer
- **Produces:** Trigger events → Descriptor FIFO
- **Responsibilities:** Decides when a pulse event has occurred. Three modes:
  - `LEADING_EDGE`: threshold crossing on rising edge
  - `CFD` (Constant Fraction Discrimination): fires at a fixed fraction of pulse amplitude, improving timing resolution
  - `ML` (optional): neural-network-based classifier running on FPGA fabric
- **Clock domain:** `sys_clk`
- **Output per event:** trigger timestamp (sample counter value), trigger type tag
- **Debug interface (§12.1):** FSM state (IDLE/DETECT/HOLDOFF), trigger_count (total accepted/rejected), holdoff_counter (live), threshold crossing rate (Hz), armed status

**HPGe‑specific notes:**
- HPGe pulse rise times are ~100–500 ns (planar detectors) or ~200–1000 ns (coaxial detectors) = **10–105 samples @ 105 MSPS**. The rising edge contains position-dependent interaction information critical for pulse-shape discrimination and ML-based event localisation.
- Leading-edge trigger threshold should be settable with fine granularity (e.g. 1 mV steps at 14-bit resolution) to handle weak pulses near the noise floor.
- CFD delay must be programmable to match the expected rise time.
- **Event window**: 1800 samples (600 pre + 1200 post) = 17.14 µs. The rising edge occupies 0.6–6% of the total window — the rest is pre-trigger baseline and post-trigger tail for energy integration. Window size is configurable: 512-sample window (51.2 µs) suffices for physics while doubling max event rate.

**Rate-limiting context:** Because the host link (FT232H) is limited to ~30 MB/s, the trigger serves as the primary rate controller. See [Throughput Budget](#throughput-budget--rate-limiting) for event rate limits.

### 6. Monitor & Statistics

The Monitor & Statistics block is now integrated into the **standardised debug architecture** (§12). It no longer requires dedicated per-signal wiring — every pipeline block exposes its debug state via `dbg_if`, and the **Audit Aggregator** (§12.4) collects everything into a unified register map.

- **Produces:**
  - **Register-mapped telemetry** via DPTI command channel: any debug counter from any block can be read by address (§12.4)
  - **Periodic diagnostic snapshots** (frame type `0x04`): the Audit Aggregator samples all debug counters once per second (or on-demand) and injects a snapshot packet into the TX FIFO
  - **Per-event diagnostic snapshot** embedded in each waveform descriptor (frame type `0x02`): coprocessed FIFO error flags, trigger rate, wr_ptr value
- **Clock domain:** `sys_clk` (with synchronisers for `adc_dco` domain counters)
- **Notes:** The diagnostic counters previously listed in a static table are now **dynamically readable** via the register map — no need to hardcode a fixed set. The host can query any block's debug state at any time, not just at snapshot intervals.

### 7. Continuous Capture

- **Receives:** Conditioned sample stream
- **Produces:** Sample blocks → Waveform Reader
- **Responsibilities:** Packages raw samples into fixed-size blocks for streaming without requiring a trigger. Used for diagnostics, baseline monitoring, or short-duration always-on recording.
- **Clock domain:** `sys_clk`
- **Limitation:** Sustained continuous capture at 105 MS/s (184 MB/s raw) **exceeds** the FT232H host link capacity (30 MB/s). Continuous mode is therefore **burst-only** — enabled for short periods (e.g. 1 second = ~180 MB, which takes ~6 seconds to drain over the host link). Use for:
  - Initial system validation / noise characterisation
  - Diagnostic recording of known short-lived phenomena
  - Baseline acquisition for trigger threshold calibration
- **Notes:** When active, competes with trigger-based readout for TX FIFO bandwidth. Arbitration logic must ensure trigger events are not starved (see Key Constraints §7).
- **Debug interface (§12.1):** FSM state, blocks_sent, bytes_sent, active flag, dropped_block_count

### 8. Descriptor FIFO

- **Receives:** Trigger events from Trigger Logic
- **Produces:** Event descriptors to TX FIFO arbitrator
- **Per-event descriptor fields (160-bit packed `descriptor_t`):**
  - `timestamp` — 48-bit, sample counter value at trigger point (wide enough to never wrap in a run)
  - `amplitude` — 16-bit, peak sample value (or CFD interpolated value)
  - `energy` — 16-bit, integrated area under pulse (if computed by Signal Processing)
  - `trigger_type` — 2-bit enum: `LEADING_EDGE | CFD | ML`
  - `waveform_offset` — 13-bit, buffer address of pretrigger start (8192-depth)
  - `waveform_length` — 11-bit, number of samples to read (up to 2048)
  - `channel_id` — 1-bit, which ADC channel triggered (0 or 1)
  - `board_id` — 16-bit, unique board identifier (hard-coded in bitfile)
  - `firmware_version` — 32-bit, bitfile version (major.minor.patch.build)
  - `diag_snapshot` — 5-bit, coprocessed status at trigger time (ADC error, CDC overflow, desc overflow, TX overflow, trigger饱和)
  - **Total: 48+16+16+2+13+11+1+16+32+5 = 160 bits**
- **Clock domain:** `sys_clk`
- **Notes:** Every waveform packet in the output stream bundles the descriptor with its waveform payload and diagnostic snapshot, creating a fully self-describing, traceable dataset element per data_engine.md §2.3.
- **Debug interface (§12.1):** fill level (instantaneous + watermark), push/pop count, overflow flag, lost_event_counter, oldest descriptor timestamp

### 9. Waveform Reader

- **Receives:** Descriptor (waveform_offset + waveform_length) from Descriptor FIFO or Continuous Capture
- **Produces:** Sample payload blocks → TX FIFO
- **Responsibilities:** Reads the circular waveform buffer at the address window recorded at trigger time. Handles wrap-around at buffer boundary.
- **Clock domain:** `sys_clk`
- **Debug interface (§12.1):** FSM state (IDLE/BURST/DONE), events_read count, remaining samples (live), backpressure stall cycles, wrap_handled count

**Critical timing constraint:** The circular buffer wraps every ~78 µs (current 8192-depth implementation @ 105 MSPS write). The Waveform Reader must extract a waveform within that window. With a 100 MHz sys_clk and a 2-cycle-per-sample burst (one 1800-sample event), read time is **36 µs** — comfortably within the 78 µs wrap window for a single event, but limiting back-to-back readout to **2 consecutive events** before overwrite risk. Multiple simultaneous read requests are handled sequentially; the TX FIFO absorbs burstiness.

**Back-to-back budget:**
| Events in FIFO | Read time | Margin consumed | Remaining |
|:--------------:|:---------:|:---------------:|:---------:|
| 1 | 36 µs | 3600 samples | 4592 samples (43.7 µs) |
| 2 | 72 µs | 7200 samples | 992 samples (9.4 µs) |
| 3 | 108 µs | **OVERFLOW** | — |

- **Notes:** Must account for the latency between trigger and readout — the write pointer will have advanced; the reader must use the snapshot address, not the live pointer. Address arithmetic must handle the BRAM word width (may pack multiple samples per word).

### 10. TX FIFO

- **Receives:** Event descriptors, waveform payloads, continuous-capture blocks, and diagnostic snapshots
- **Produces:** Ordered byte stream to FT232H bridge
- **Responsibilities:** Elastic buffer to absorb burst traffic; arbitrates between data sources. Emits a framed byte stream matching the packet format defined in §Host Interface.
- **Clock domain:** `sys_clk` (write side); `ft_clk` (60 MHz, read side) — requires a second CDC here
- **Key parameters:**
  - `TX_FIFO_DEPTH` — sized to absorb worst-case burst (e.g. one full waveform + descriptor). A 2048×32 dual-clock FIFO is sufficient.
  - Output width: **8 bits** (matches FT232H data bus width)
- **Arbitration policy:** Strict priority: diagnostic frames > trigger events > continuous-capture blocks. Within each class, round-robin.
- **Debug interface (§12.1):** fill level (instantaneous + watermark), overflow/underflow flags, words_written/read counts, dpti_stall_cycles

### 11. FT232H DPTI Host Interface

- **Receives:** 8-bit byte stream from TX FIFO
- **Produces:** USB 2.0 High-Speed bulk transfers to host PC (via on-board FT232H)
- **Protocol:** DPTI (Digilent Parallel Transfer Interface) — synchronous FIFO mode
- **Clock domain:** `ft_clk` (60 MHz from FT232H `prog_clko`)
- **Debug interface (§12.1):** TX stall count (prog_txen high cycles), RX command count, bus turnaround count, protocol error count, bytes_sent/bytes_read

**FT232H pinout (on-board, Bank 14 3.3V):**

| FPGA Signal | FPGA Pin | Direction | Description |
|-------------|:--------:|:---------:|-------------|
| `prog_clko` | P17 | FT232H → FPGA | 60 MHz clock output |
| `prog_d[7:0]` | M18, R12, R13, M13, R18, T18, N14, P14 | Bidir | 8-bit data bus |
| `prog_txen` | M17 | FT232H → FPGA | TX FIFO not full (active low) — FPGA must check before writing |
| `prog_rxen` | M16 | FT232H → FPGA | RX FIFO has data (active low) — FPGA may read |
| `prog_wrn` | N16 | FPGA → FT232H | Write strobe (active low), 1 cycle = 1 byte written |
| `prog_rdn` | N15 | FPGA → FT232H | Read strobe (active low), 1 cycle = 1 byte read |
| `prog_oen` | N17 | FPGA → FT232H | Output enable (active low) — FPGA drives bus when asserted |

**Synchronous FIFO Write (FPGA → PC):**

```
1. Wait for prog_txen == 0  (FT232H TX FIFO has space)
2. Set prog_oen = 0          (FPGA drives data bus)
3. Drive prog_d[7:0] with data byte
4. Assert prog_wrn = 0 for one rising edge of prog_clko
5. Repeat for next byte
```

**Synchronous FIFO Read (PC → FPGA):**

```
1. Wait for prog_rxen == 0  (FT232H RX FIFO has data)
2. Set prog_oen = 1          (FT232H drives data bus)
3. Assert prog_rdn = 0 for one rising edge of prog_clko
4. Sample prog_d[7:0] on next rising edge
```

**Throughput:** Maximum sustained rate is ~30 MB/s (limited by USB 2.0 HS bulk transfer + FT232H overhead). Peak burst rate can approach 60 MB/s for short periods.

**DPTI Control Channel:** In addition to the data stream, the DPTI interface supports a command/response channel for register access (read/write audit counters, configure trigger thresholds, start/stop acquisition). Commands are multiplexed onto the same bidirectional data bus using a type-length-value (TLV) scheme.

## 12. Debug & Validation Architecture

Every pipeline block exposes a **standardised debug interface** that the **Audit Aggregator** collects onto a DPTI-accessible register map. This makes every internal state observable from the host without JTAG or probing — the FPGA debugs itself.

### 12.1 Standard Debug Interface (dbg_if)

Each module defines a `dbg_if` output port (SystemVerilog interface or packed struct). The interface is **optional to connect** — unused debug ports are synthesis-trimmed, adding zero cost when the aggregator is not instantiated.

```systemverilog
// 128-bit packed debug struct — common to all blocks
// 32-bit-aligned for clean 4-register access via Audit Aggregator
// Field layout matches docs/debug_register_map.md
typedef struct packed {
    // --- Reg+0x00 (bits 127:96) ---
    logic [3:0]  state;           // [127:124] FSM state (0 = IDLE)
    logic [15:0] cycle_count;     // [123:108] cycles since reset
    logic [7:0]  stall_cycles;    // [107:100] cycles waiting on backpressure
    logic [3:0]  reserved_0;      // [99:96]   reserved (zero)

    // --- Reg+0x04 (bits 95:64) ---
    logic        error;           // [95]    sticky error flag
    logic [7:0]  error_id;        // [94:87] which error fired
    logic        bypass_mode;     // [86]    is this block bypassed?
    logic [21:0] reserved_1;      // [85:64] reserved (zero)

    // --- Reg+0x08 (bits 63:32) ---
    logic [15:0] event_count;     // [63:48] events processed by this block
    logic [15:0] reserved_2;      // [47:32] reserved (zero)

    // --- Reg+0x0C (bits 31:0) ---
    logic [15:0] word_count;      // [31:16] words consumed/produced
    logic [15:0] reserved_3;      // [15:0]  reserved (zero)
} dbg_info_t;                     // Total: 128 bits = 4 × 32-bit registers

// Interface wrapper (carries clock + packed debug)
interface dbg_if;
    logic        clk;             // module's clock domain
    dbg_info_t   info;            // 128-bit packed debug data
    // Block-specific signals exposed separately (mapped to reg+0x10..+0x1C)
    // See docs/debug_register_map.md for per-block field assignment
endinterface
```

**Synthesis semantics:** When the `dbg_if` port is unconnected, the synthesiser optimises away the debug logic (the `export dbg_if` output is tied off by the parent, and the debug counters are never toggled). This ensures zero area cost when debug is compiled out.

### 12.2 Sample Continuity Token

Alongside each data sample through the pipeline, propagate a **continuity token** that encodes a monotonic sample sequence ID:

```systemverilog
typedef struct packed {
    logic        valid;     // [52] data valid
    logic [15:0] sample;    // [51:36] the actual ADC sample (16-bit, zero-extended from 14-bit)
    logic [31:0] seq_id;    // [35:4]  sample sequence ID (monotonic, 32-bit — no wrap in 41s @ 105 MSPS)
    logic [3:0]  flags;     // [3:0]   metadata flags
} sample_token_t;           // Total: 53 bits
```

Every pipeline stage passes `seq_id` through unchanged. At the **TX FIFO** output (or at any inter-block boundary), check that `seq_id` increments by exactly 1 each valid cycle. A gap means a sample was dropped somewhere upstream. This catches:

- CDC FIFO overflow (writes lost crossing clock domains)
- BRAM address misalignment during wrap-around read
- Pipeline stall that misaligns data
- Glitch filter logic that inadvertently swallows a valid sample

The host can poll the audit aggregator for `word_count` on each block and compare — a mismatch between `ADC_Interface.word_count` and `CDC_FIFO.word_count` pinpoints exactly which stage lost data.

### 12.3 Per-Block Debug Capabilities

| # | Block | `state` | `error` conditions | `event_count` | `stall_cycles` | Notes |
|:-:|-------|---------|--------------------|:-------------:|:--------------:|-------|
| 1 | ADC Interface | IDDR lock, deinterleave | DCO missing, bit error | Samples captured | N/A (source) | Count resets on DCO loss |
| 2 | CDC FIFO | fill_level[5:0] | overflow, underflow | Words crossed | Read-empty stalls | Watermark trackable via `word_count` diff |
| 3 | Glitch Filter | BYPASS / ACTIVE | Filter saturation | Glitches removed | N/A (pipeline) | Max delta (max jump observed) also exposed |
| 4 | Circular Buffer | WRITE / BURST_READ | wr_ptr == rd_addr (collision) | Wrap_around count | Burst-read stalls | Live wr_ptr and rd_addr readable |
| 5 | Trigger | IDLE / DETECT / HOLDOFF | Threshold saturation | Triggers fired | Holdoff wait | Also exposes: threshold_cross_rate, armed |
| 6 | Continuous Capture | IDLE / STREAM | Overflow (FIFO full) | Blocks_sent | TX FIFO full stalls | — |
| 7 | Descriptor FIFO | fill_level[5:0] | overflow, seq gap | Descriptors pushed | Full stalls | Driven by trigger event_count |
| 8 | Waveform Reader | IDLE / BURST / DONE | length mismatch, wrap error | Events read | TX FIFO full stalls | Remaining count live-visible |
| 9 | TX FIFO | fill_level[11:0] | overflow, underflow | Words drained | DPTI stall | Dual-clock: stats synced to sys_clk |
| 10 | DPTI Bridge | IDLE / SEND | protocol error, timeout | Bytes sent | TXE# wait | Also exposes: rx_cmd_count, bus_turnarounds |

### 12.4 Audit Aggregator & Register Map

The **Audit Aggregator** is a central module that:

1. **Collects** all `dbg_if` ports from every pipeline block
2. **Maps** them to a flat register address space (16-bit address, 32-bit data)
3. **Serves** register reads/writes via the DPTI command channel
4. **Periodically** snapshots all debug counters into a diagnostic frame (type `0x04`) injected into the TX FIFO

**Register map (base addresses, 0x20 spacing per block = 8 × 32-bit registers):**

| Address Range | Block | Readable Fields |
|:-------------:|-------|-----------------|
| `0x000`–`0x01F` | System | firmware_version, board_id, run_ts, reset_cause, sys_ctrl |
| `0x020`–`0x03F` | ADC Interface (1) | state, error, event_count, word_count, chA_sync, chB_sync, iddr_cal_state |
| `0x040`–`0x05F` | CDC FIFO (2) | state, error, event_count, word_count, fill_level, watermark |
| `0x060`–`0x07F` | Glitch Filter (3) | state, error, event_count (glitches), max_delta, bypass, threshold |
| `0x080`–`0x09F` | Circular Buffer (4) | state, error, event_count (wraps), wr_ptr, rd_ptr, collision, watermark |
| `0x0A0`–`0x0BF` | Trigger (5) | state, error, event_count, threshold_cross_rate, armed, holdoff_remaining |
| `0x0C0`–`0x0DF` | Continuous Capture (6) | state, error, event_count (blocks), active, dropped_block_count |
| `0x0E0`–`0x0FF` | Descriptor FIFO (7) | state, error, event_count (pushed), fill_level, watermark, lost_event_count |
| `0x100`–`0x11F` | Waveform Reader (8) | state, error, event_count, stall_cycles, remaining, wrap_handled |
| `0x120`–`0x13F` | TX FIFO (9) | state, error, event_count (drained), fill_level, watermark, dpti_stall |
| `0x140`–`0x15F` | DPTI Bridge (10) | state, error, bytes_sent, stall_cycles, rx_commands, bus_turnarounds |
| `0x1E0`–`0x1FF` | BIST control | bist_mode, bist_pattern, bist_error_mask, bist_result |

**Full field-level register map** (16-bit address, 32-bit data, bit offsets, R/W, reset values, descriptions) is in `docs/debug_register_map.md`. This is the authoritative contract between RTL, Audit Aggregator, and host software. Each block has 8 registers: 4 common (`dbg_info_t` at +0x00..+0x0C) + 4 block-specific (+0x10..+0x1C).

**Host read/write protocol (DPTI command channel):**

| CMD Byte | Direction | Effect |
|:--------:|:---------:|--------|
| `0x10 + addr_hi` | Host→FPGA | Set register address high byte (next read/write targets this addr) |
| `0x11 + addr_lo` | Host→FPGA | Set register address low byte |
| `0x12 + data_hi` | Host→FPGA | Write data high byte to selected address |
| `0x13 + data_lo` | Host→FPGA | Write data low byte to selected address |
| `0x14` | Host→FPGA | Read request — FPGA replies with 4 bytes of register data |
| `0x15` | FPGA→Host | Read response (4 data bytes) |

This replaces the ad-hoc command set (current: query_channel, query_glitch) with a uniform, extensible register interface. Adding new debug observability requires only wiring the block's `dbg_if` to the aggregator — no command decoder changes.

### 12.5 BIST (Built-In Self-Test)

A **BIST mode** validates the entire FPGA pipeline without any external equipment (no ADC, no detector, no pulse generator).

**Activation:** Write `bist_mode = 1` to register `0xF0` via DPTI.

**What happens:**

1. A **pattern generator** (LFSR or linear ramp) replaces the ADC data source at the pipeline input
2. Each pipeline block compares its output against the expected pattern internally and sets `dbg.error` on mismatch
3. The Audit Aggregator collects all error flags
4. The host polls register `0xF4` (bist_result) — a bitmask: bit N = 1 means block N detected an error
5. The host can narrow down: set `bist_pattern = ramp` (0xF1), run N events, read each block's `event_count` to see how far data propagated before an error

**BIST patterns:**

| Pattern ID | Description | Catches |
|:----------:|-------------|---------|
| `0x00` | Linear ramp 0..65535 (wrapping) | BRAM addressing, pointer wrap, pipeline alignment |
| `0x01` | Alternating max/min (0, 65535, 0, ...) | CDC FIFO full-word transfer, glitch filter bypass |
| `0x02` | Single pulse with known rise time | Trigger threshold, event window sizing |
| `0x03` | PRBS (LFSR-16) | Statistical: data integrity under backpressure |
| `0x04` | Glitch-injected ramp (ramp + 0x0C98 every 256) | Glitch filter detection rate, false positives |

**BIST pass criteria:** All blocks report `error = 0` after 1,000,000 pattern cycles at `bist_pattern = ramp`. Failure means a hardware bug in the RTL — caught before connecting a real detector.

### 12.6 Host-Side Debug Tools

The host software (`collect_waveforms_root.py` or its successor) provides:

- **`debug_poll_all()`** — reads all register map entries, prints a table of every block's state, event_count, and error flags. Takes ~100 µs over DPTI. Non-intrusive during acquisition.
- **`debug_poll_block(addr)`** — reads a single block's debug registers. Use for targeted investigation.
- **`debug_bist(pattern_id, cycles)`** — runs BIST mode for N cycles, polls result, prints pass/fail per block.
- **`debug_continuity_check()`** — reads `word_count` from ADC Interface and TX FIFO, reports if they match. A mismatch identifies the first block where `word_count` diverges.
- **`debug_watermark_check()`** — reads all FIFO watermark registers, prints max fill levels observed. Useful for sizing FIFOs.

| What                     | Format                              | Path                                                         |
|--------------------------|-------------------------------------|--------------------------------------------------------------|
| Raw samples              | 14-bit × 2 ch @ 105 MS/s            | ADC → Interface → CDC → Glitch filter → Processing → Buffer                 |
| Triggered event          | Descriptor (metadata only)          | Trigger Logic → Descriptor FIFO → TX FIFO → FT232H           |
| Triggered waveform       | Descriptor + sample array + diag    | Descriptor FIFO + Waveform Reader → TX FIFO → FT232H         |
| Continuous stream        | Fixed-size sample blocks (burst)    | Continuous Capture → Waveform Reader → TX FIFO → FT232H      |
| Monitor telemetry        | Register reads (I²C or DPTI cmd)    | Monitor & Stats → host (separate control channel)            |
| Monitor provenance       | Diagnostic snapshot every N events   | Monitor & Stats → Descriptor FIFO → TX FIFO → FT232H         |
| Host commands            | TLV-encoded register R/W            | FT232H → TX FIFO readback → DPTI command decoder             |

## Host Interface & Packet Format

The FT232H DPTI bridge streams a sequence of **frames** to the host. Each frame is self-describing (length + type tag) so the host software can reassemble and interleave different data types without out-of-band signalling.

### Frame Types

| Type Tag  | Payload                                      | Source                         | Frequency                        |
|-----------|----------------------------------------------|--------------------------------|----------------------------------|
| `0x01`    | Trigger descriptor (metadata only)           | Descriptor FIFO                | Per trigger event                |
| `0x02`    | Waveform payload (descriptor + sample array) | Waveform Reader                | Per triggered waveform           |
| `0x03`    | Continuous sample block                      | Continuous Capture             | Burst-only                       |
| `0x04`    | Diagnostic snapshot                          | Monitor & Stats (provenance)   | Periodic (e.g. every 1 s)        |
| `0x05`    | Command response                             | DPTI command decoder            | Per host command                 |
| `0xFF`    | Run header / footer                          | Host command → FPGA reply      | Start / end of run               |

### Common Frame Header (32 bytes)

| Offset | Size   | Field               | Description                                      |
|--------|--------|----------------------|--------------------------------------------------|
| 0      | 2      | magic                | `0xDA7A` (Data Acquisition frame marker)         |
| 2      | 2      | frame_type           | One of the type tags above                       |
| 4      | 4      | frame_length         | Total frame length in bytes (including header)   |
| 8      | 8      | run_timestamp        | 64-bit run time counter (nanoseconds since run start) |
| 16     | 2      | board_id             | Board identifier (matches descriptor field)      |
| 18     | 4      | firmware_version     | Bitfile version (matches descriptor field)       |
| 22     | 2      | sequence_number      | Monotonically increasing sequence per frame type |
| 24     | 8      | reserved             | Future use (zero‑filled)                         |

### Host-side conversion

The host acquisition software (Python / C++) parses the frame stream from the FT232H and converts it into HDF5 or ROOT files:

```
/run_metadata         → Attributes: board_id, firmware_version, run_start_time, ADC_config
/waveforms            → Dataset: table with columns [timestamp, energy, trigger_type, channel, waveform_samples]
/diagnostics          → Dataset: table with columns [timestamp, adc_errors, overflow_flags, trigger_rate, ft_stalls]
/continuous           → Dataset: raw sample blocks (optional, burst-only)
/configuration        → Snapshot of all register settings at run start
```

This structure fulfils the dataset provenance requirements described in data_engine.md §2.3 and §6.

## Throughput Budget & Rate Limiting

The on-board FT232H is the system's primary bottleneck. All design decisions must account for its throughput limits.

### Empirical vs. Theoretical Throughput

| Measure | Value | Notes |
|---------|:-----:|-------|
| **FT232H theoretical (USB 2.0 HS)** | ~30 MB/s | Burst-limited by USB bulk transfer |
| **FT232H sustained (empirical)** | **~7 MB/s** | Measured with DPTI round-robin PHY |
| **ADC raw (1 ch, 14-bit @ 105 MS/s)** | 184 MB/s | 26× bottleneck vs. empirical |
| **ADC raw (2 ch)** | 368 MB/s | 53× bottleneck |

### Raw vs. Selected Throughput

| Mode | Data Rate | Feasible @ 7 MB/s? | Feasible @ 30 MB/s? |
|---|---|---|---|
| **Continuous, 1 ch, 105 MS/s** | 14 b × 105 M = **184 MB/s** | ❌ No | ❌ No |
| **Continuous, 2 ch** | 2 × 184 = **368 MB/s** | ❌ No | ❌ No |
| **Triggered, descriptor-only** | ~48 B/evt × 625K evt/s = **30 MB/s** | ❌ Exceeds | ✅ At limit |
| **Triggered, 256-samp waveform** | ~560 B/evt × 53K evt/s = **30 MB/s** | ❌ Exceeds | ✅ At limit |
| **Triggered, 512-samp waveform** | ~1,072 B/evt × 28K evt/s = **30 MB/s** | ❌ Exceeds | ✅ At limit |
| **Triggered, 1024-samp waveform** | ~2,096 B/evt × 14K evt/s = **29 MB/s** | ❌ ~3.3K evt/s cap | ✅ At limit |
| **Triggered, 1800-samp waveform** | ~3,632 B/evt × 8.3K evt/s = **30 MB/s** | ✅ ~**1,900 evt/s** cap | ✅ At limit |
| **Triggered, 4096-samp waveform** | ~8,192 B/evt × 3.6K evt/s = **30 MB/s** | ✅ ~**850 evt/s** cap | ✅ At limit |

**Empirical rule of thumb:** `max_event_rate × (descriptor_size + waveform_size) ≤ 7 MB/s`

The host software can dynamically adjust the requested waveform length based on observed trigger rate to prevent buffer overflow. For HPGe at typical 1–5 kHz rates, a 512–1024 sample window provides full pulse shape with 2–5× headroom.

### Impact on Audit

- FT232H **stall counters** (`prog_txen` high time) are the most important diagnostic for rate management.
- A rising `DESCRIPTOR_FIFO.overflow` coincident with rising `FT232H.stall_count` indicates the host cannot keep up.
- A rising `CIRCULAR_BUFFER.wrap_around` coincident with rising `DESCRIPTOR_FIFO.overflow` indicates event loss even before the host link — the trigger is too fast for the buffer.

## Key Design Constraints

1. **No data loss:** The circular buffer must be deep enough that the write pointer never overtakes an in-progress read. Size as: `BUF_DEPTH > max_readout_latency_cycles + PRETRIGGER_SAMPLES`. With ~300K samples at 105 MS/s, the readout window is ~2.9 ms — the Waveform Reader must drain within this.

2. **Timestamp accuracy:** The trigger timestamp must be latched in the same cycle as the trigger fires. Any pipeline latency in Signal Processing must be subtracted.

3. **CDC correctness:** All cross-domain paths must be verified in timing closure. Mark unconstrained paths explicitly — do not rely on default analysis.

4. **Backpressure:** TX FIFO full must propagate backpressure to Descriptor FIFO and Waveform Reader. Overflow must set an error flag readable by host, not silently drop data.

5. **Trigger mode is runtime-configurable:** Switching trigger mode mid-run must flush the Descriptor FIFO cleanly.

6. **Diagnostic completeness:** Every waveform packet must carry a diagnostic snapshot so that datasets are self-describing and traceable without external metadata files.

7. **Continuous-capture vs. trigger mode arbitration:** When both modes are active, the TX FIFO arbitrator must implement fairness to prevent trigger events from being starved by bulk continuous streaming, or vice versa. Continuous capture is inherently burst-limited.

8. **FT232H protocol compliance:** All DPTI timing (setup/hold on `prog_wrn`/`prog_rdn`, bus turnaround for `prog_oen`) must be strictly observed. Violations cause corrupted transfers.

9. **Rate management:** The host must monitor `prog_txen` stalls and dynamically reduce event rate by raising thresholds or skipping waveforms in descriptor-only mode.

10. **Event window is runtime-configurable:** `cfg_pre` and `cfg_post` determine the event window size (default 600 + 1200 = 1800 samples = 17.14 µs). Window size directly controls max event rate over the FT232H bottleneck. For HPGe, the 200 ns rising edge (~21 samples) is the physics-critical region — 512-sample windows suffice for most ML applications while tripling throughput vs. 1800-sample windows.

11. **Glitch filter must be on the real-time path:** The 3-stage up-down pattern detector sits between CDC FIFO and downstream processing. It adds 2 cycles of pipeline latency, which must be accounted for in trigger timestamp generation. Separate `always @(posedge clk)` blocks per stage are required to prevent Vivado SRL32 inference.

12. **Standard debug interface per block:** Every pipeline module must expose a `dbg_if` output port. The Audit Aggregator is the single point of collection. See §12.

13. **BIST before detector:** Before connecting an HPGe detector or pulse generator, run BIST mode (§12.5) to validate the full pipeline from pattern generator to TX FIFO output. Do not proceed to detector operation if any block reports an error.

14. **Sample continuity invariant:** The `seq_id` in the sample continuity token (§12.2) must increment monotonically through the pipeline. Any gap detected at the Audit Aggregator must set an error flag and be reported in the next diagnostic snapshot.

| Term | Definition |

| **AD9648** | Analog Devices dual 14-bit ADC, 105 MSPS, used on the Zmod ADC 1410-105 |

| **CDC** | Clock Domain Crossing |
| **CFD** | Constant Fraction Discrimination — a triggering technique that fires at a fixed fraction of pulse amplitude, making trigger time independent of pulse height |
| **CMOS mode** | ADC output mode where data pins use single-ended CMOS levels (vs. LVDS) |
| **DCO** | Data Clock Output — clock generated by the ADC to strobe data into the FPGA |
| **Descriptor** | Metadata packet describing a single trigger event (timestamp, amplitude, energy, type) |
| **DPTI** | Digilent Parallel Transfer Interface — synchronous FIFO protocol over FT232H |
| **FT232H** | FTDI chip providing USB 2.0 High-Speed to synchronous FIFO bridge |
| **HPGe** | High-Purity Germanium — semiconductor radiation detector |
| **IDDR** | Input Double Data Rate — Xilinx primitive that captures data on both clock edges |
| **Interleaved CMOS** | ADC mode where both channels share one data bus (Ch A on rising edge, Ch B on falling edge) |
| **LVCMOS18** | 1.8 V single-ended I/O standard used by the Zmod ADC SYZYGY interface |
| **Pole-zero correction** | Digital filter that cancels the exponential decay tail added by detector shaping amplifiers |
| **Pretrigger samples** | Samples stored before the trigger point, captured because the circular buffer is always writing |
| **prog_*** | Prefix for FT232H DPTI interface signals (`prog_clko`, `prog_d`, `prog_txen`, etc.) |
| **SSTL18** | 1.8 V Stub Series Terminated Logic — differential I/O standard for the ADC sample clock output |

| **USB104 A7** | Digilent carrier board with Xilinx Artix-7 XC7A100T FPGA |
